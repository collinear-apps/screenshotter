# Improvement roadmap

Research-backed, prioritized improvements for screenshotter toward even more reliable
one-shot 1:1 functional rebuilds. Generated from a multi-agent research pass (34 grounded
proposals across 5 angles), verified against the current code.

## Headline

Screenshotter is a genuinely feature-rich capture-to-rebuild pipeline (DOM, tokens, a11y goldens, API fixtures, entity graph, explorer, dark/breakpoint variants, mock server, scaffold, two existing gates), but it is missing the single thing the north star most depends on: a closed verification loop. Verified against source, there is NO pixel/perceptual diff anywhere, and the verify/vdiff/score idea was proposed three separate times — that one unified gate is the highest-leverage build. The second cluster of cheap wins is making already-captured-but-orphaned data actually reach the rebuild agent (variant screenshots, the dead entityGraph param, the dead ComponentEntry type, MCP return paths).

## Top recommendations

### 1. Unified verify/score command with pixel + a11y + functional + data gates  _(impact: high · effort: M)_

- **What:** Single `verify <bundle> --target <url>` (folding in the proposed vdiff/score) that re-captures the rebuild with the existing deterministic context, runs pixelmatch+pngjs perceptual diff per route, loops a11y-diff over every .aria.yaml golden, runs runQcTasks, and emits one weighted fidelity score + verify-report.json with non-zero exit and worst-route ranking.
- **Why:** Confirmed: zero pixel/perceptual diff exists (no pixelmatch/pngjs/odiff in src or package.json), the determinism layer already advertises diff-readiness with no consumer, and a11y verification is currently a hand-written bash for-loop in the prompt. This is the one signal humans grade on and the only way a one-shot rebuild can self-correct. Three of the proposals (vdiff, verify, score) are the same feature — build it once.

### 2. Record breakpoint/dark variant screenshots in bundle.json  _(impact: medium · effort: S)_

- **What:** In assembleRouteArtifacts, populate screenshots[] with all already-on-disk @tablet/@mobile/@dark variants tagged {path,breakpoint,scheme} instead of just [shot].
- **Why:** Verified: bundle.ts line 71 sets screenshots:[shot]; the variant PNGs are written to disk by the pipeline but orphaned — invisible to scaffold, prompt, and any future diff gate. Tiny change that unlocks responsive/dark fidelity for free and is a prerequisite for the verify gate to diff variants.

### 3. Activate the dead entityGraph param + listing values in the data-fidelity gate  _(impact: high · effort: S)_

- **What:** Add buildEntityDataTasks/buildListingValueTasks in qc/generate.ts mining salient values from EntityGraph.entities and ListingExtract.rows; the runner already handles arbitrary expectText.
- **Why:** Verified: entityGraph is on QcBuildInput (line 388) but never consumed — buildAllQcTasks only feeds fixtures to buildDataTasks. SSR/server-rendered sites expose little XHR, so today they get near-zero data assertions and a placeholder-data rebuild passes. Captured data already exists and is loaded; this just wires it. Pure win, no runner change.

### 4. Element interaction-state capture (hover/focus/active/disabled) — fix the wired no-op  _(impact: high · effort: M)_

- **What:** Build the state-capture pass keyed off cfg.elementStates: drive Playwright pseudo-states on a deduped per-region control sample, emit tight clipped PNGs and (load-bearing) element-states.json with computed-style DELTAS resting→hover/focus/active.
- **Why:** Verified: cfg.elementStates is declared and defaulted true but has ZERO consumers — a dead flag promising a feature that doesn't run. Hover colors, focus rings, and disabled opacity are among the most visible interactivity tells; the computed-style deltas give exact token values instead of a fuzzy screenshot read.

### 5. Pierce Shadow DOM in DOM/token/clickable capture  _(impact: high · effort: M)_

- **What:** Recurse into open .shadowRoot and adoptedStyleSheets in captureDom (serialize as declarative shadow DOM), extractTokens, and enumerateClickables; emit web-components.json inventory.
- **Why:** Verified: zero shadowRoot/adoptedStyleSheets references in src; captureDom is plain page.content() (light DOM only). Any web-component/design-system site (Lit, LWC, embedded widgets) currently yields empty shells — no structure, no scoped CSS, unreachable controls. Increasingly common and currently a total blind spot.

### 6. Emit a typed API client + TS types from the already-inferred schemas  _(impact: high · effort: M)_

- **What:** Codegen scaffold/src/api/types.ts (interfaces per endpoint) and a typed client (listModels(query): Promise<...>) from the existing inferSchema/mergeSchemas + catalog; optionally zod (already a dep).
- **Why:** Verified: real merged JSON Schemas are produced but never compiled; the only emitted client is apiGet<T=unknown>. The prompt tells the agent to 'build a typed client' three times while shipping an untyped one, forcing per-shape re-derivation and data-shape drift. The schemas are sitting right there.

### 7. Bot-wall / CAPTCHA interstitial detection and quarantine  _(impact: high · effort: M)_

- **What:** Add detectChallenge(page) alongside detectAuthState: probe for Cloudflare/hCaptcha/PerimeterX signatures + suspiciously-thin DOM on rich routes; mark challenged in manifest, exclude from token/entity aggregation, warn, optional one retry.
- **Why:** Verified: gotoWithRetry treats 200 as success regardless of content and detectAuthState has no challenge branch. A challenge page is captured as if it were the site and marked ok=true — silently poisoning the bundle so the rebuild clones a Cloudflare splash. Converts silent corruption into an actionable signal.

### 8. First automated test suite over the pure load-bearing modules  _(impact: high · effort: M)_

- **What:** Add node:test/vitest + a test script covering retry backoff/Retry-After parsing, crawler normalize(), entity-graph inference (incl. the spurious-owner cases), fixture canonicalJson/stableHash/querySignature/stabilizeBody, redaction.
- **Why:** Verified: package.json has only build/typecheck/start, no test files anywhere (one module even has an 'Exposed for tests' comment with no test). These pure functions silently corrupt the bundle on regression — the exact failure a reviewer won't catch and that breaks a one-shot rebuild. Foundational safety net for everything else.

## Tiers

### Tier 1 — Quick wins (S effort, ship first)

_Verified-cheap changes that wire up data already captured but currently dropped on the floor, or close obvious gaps. Highest impact/effort ratio; several are prerequisites for the verify gate._

- Activate the dead entityGraph param + listing-row values in the data-fidelity gate (qc/generate.ts) — biggest S-effort win, unblocks SSR sites.
- Record breakpoint/dark variant screenshots in bundle.json (assembleRouteArtifacts sets only [shot] today) — prerequisite for diffing variants.
- MCP tool parity + structured return paths: return rebuildPromptPath/bundleIndexPath/mockServerPath/manifestPath + quality score (pipeline.run already returns these; the MCP layer discards them).
- Per-route acceptance-criteria block in REBUILD-PROMPT from BundleRouteArtifacts (already-computed linkage, prompt only uses the flat pages[]).
- Emit framework theme (tailwind.config) + semantic token names from DesignTokens — but sequence after/with the component+token work to avoid double naming passes.

### Tier 2 — Verification loop (the north-star gate)

_The single biggest missing capability. DEDUP NOTE: the proposals 'vdiff', 'verify command', and 'scorecard/score' are the SAME feature proposed three times — build ONE unified command. Without this the rebuild loop cannot self-correct._

- Build one `verify <bundle> --target <url>` that runs functional (runQcTasks) + structural (a11y-diff over all goldens) + NEW pixel/perceptual diff (pixelmatch+pngjs) and emits a single weighted fidelity score, verify-report.json, worst-route ranking, diff PNGs, masked dynamic regions, non-zero exit.
- Reuse routeUrlFor, planCaptures, captureScreenshot, determinism.ts, and --mask for capture parity.
- Simplify the prompt's Verify section to this one command (removes the hand-written bash for-loop over *.aria.yaml).
- Diff each recorded variant screenshot at its viewport once variant-screenshots (Tier 1) lands.

### Tier 3 — Capture fidelity (close the blind spots)

_Concrete cases where the rebuild gets a blank box or default styling because capture never saw the truth. Verified absent in source. Rank within tier by breadth of sites affected._

- Element interaction-state capture (hover/focus/active/disabled) — fix the cfg.elementStates dead flag; computed-style deltas are the load-bearing artifact.
- Pierce Shadow DOM + web components in DOM/token/clickable capture.
- Exact box-model/layout geometry capture (layout.json) for ~20-40 landmarks per page/breakpoint — pins spacing to structure, the hardest thing to match by eye.
- Dynamic surfaces (canvas/video/WebGL/iframe) as typed first-class artifacts (surfaces.json) with rendered thumbnails + charting-lib sniff.
- Loading/empty/error/skeleton states as first-class screenshot+DOM artifacts (promote the already-detected sampleTransientState + induce via route interception / no-match queries).
- Coverage-aware explorer enumeration (viewport-stratified scan replacing first-1500, per-region dedupe quota, interactive-density metric) — fixes the known 4-actions-on-busy-page gap.
- i18n/locale + RTL + print-media passes (reuse the existing isolated-context variant machinery; low effort).

### Tier 4 — Rebuild-spec richness (componentization + typed contracts)

_Make the agent build the RIGHT abstractions once instead of re-deriving per route. NOTE: 'ComponentEntry population', 'component inventory', and 'design-system/component-library extraction' overlap heavily — converge on ONE cross-page structural-clustering pass that populates the dead ComponentEntry type and emits components.json + variant states._

- One cross-page component-inventory pass (generalize listings.ts clustering) that populates the dead ComponentEntry type, emits components.json with routes/region/representative snippet + token usage + observed variants, links from bundle.json, and feeds a 'build shared components first' prompt step. Element-state capture (Tier 3) supplies the variant states.
- Typed API client + TS types (+ optional zod) generated from the already-inferred JSON Schemas — replaces apiGet<unknown>.
- Optional --label LLM pass (Claude API — load claude-api skill for model ids) for component naming, page archetype, and control-intent labels, cached by content hash. Lower priority: heuristic component inventory should land first; the LLM pass enriches it.

### Tier 5 — Scale & robustness (correctness at 150 pages)

_Prevents silent bundle corruption and wasted work on large crawls. The test suite is foundational and should arguably move earlier; the rest are robustness multipliers._

- First automated test suite over pure modules (retry/normalize/entities/fixtures/redact) — foundational; consider pulling into Tier 1.
- Bot-wall/CAPTCHA detection + quarantine (detectChallenge) — prevents cloning a Cloudflare splash.
- Capture-quality scoring: flag blank/soft-404/partially-rendered routes, exclude from aggregation, --min-quality gate.
- Fix entity-graph spurious-owner inference with type-aware owner validation (docs/index paths minting fake users) — small, pairs with the entity-graph tests.
- Incremental/resumable capture with per-route content hashing (--incremental/--resume) — L effort; high value only for heavy re-run workflows.
- Deeper value-stabilization (counts/UUIDs/relative-time) across DOM+fixtures — directly improves verify-gate signal-to-noise; sequence near Tier 2.
- Adaptive per-host rate-limit governor (AIMD + per-host concurrency + header-driven backoff).

### Tier 6 — DX, distribution & ambitious leverage

_Adoption and differentiation. Config/presets and CI gating make the tool reusable; the novel multi-axis captures are high-ceiling but L-effort — pursue after the verification loop and fidelity gaps are solid._

- screenshotter.config.{json,ts} + named presets + per-host overrides (single funnel via BuildConfigInput; also unblocks MCP parity).
- GitHub Action + --report json + --fail-on gating (no .github/ exists; pipeline exits 0 regardless today).
- `diff <bundleA> <bundleB>` capture-vs-capture drift monitor (reuse a11y/diff.ts + design-tokens.json).
- Unified live progress/ETA for the main capture loop (extract the explorer's existing bar helper).
- Layout-similarity clustering → templates.json + --sample-per-template (big time savings on large crawls; L effort).
- Declarative profiles.json + `profiles` listing command.
- AMBITIOUS: Multi-step user-flow capture → replayable flow scripts + a `flow` qc task kind (L; closes 'all parts work' vs 'the app works').
- AMBITIOUS: Auth/role MATRIX capture (per-persona subtrees + role-diff + role-gating qc) — high product value, M-L effort, depends on solid single-session capture first.

## Full proposal set (34)

### Angle: capture-fidelity

#### Implement element interaction-state screenshots (hover/focus/active/disabled) — the wired no-op  _(impact: high · effort: M)_

- **Problem:** A rebuilt twin's interactive controls (buttons, links, inputs, cards) render with default resting styles only. There is no captured ground truth for what a control looks like on hover, focus, focus-visible, active, or disabled — so the agent guesses hover colors, focus rings, and disabled opacity, which are some of the most visible interactivity tells of a real UI.
- **Proposal:** Build a real state-capture pass keyed off the existing `cfg.elementStates` flag. For a bounded, deduped sample of representative controls per page (reuse `enumerateClickables` signatures so a list of 50 cards yields ~1 representative), drive Playwright pseudo-states and capture a TIGHT clipped screenshot of each: `locator.hover()` for :hover, `locator.focus()` for :focus/:focus-visible, `page.mouse.down()` held over the element for :active, and natively-disabled controls for :disabled. Additionally diff the element's computed style (color/background/border/box-shadow/outline/transform/opacity) between resting and each state via getComputedStyle, and emit a `element-states.json` recording only the DELTAS (resting → hover/focus/active). Write small clipped PNGs under `<mode>/states/<page>/` and a markdown summary. The computed-style delta is the load-bearing artifact: it gives the rebuild exact hover/focus token values instead of a fuzzy screenshot read.
- **Why it matters:** Interaction-state styling is what makes a twin feel alive vs. dead. Today it is explicitly declared (types.ts:347, config.ts:309 default true) but consumed nowhere — a literal no-op the prompt itself flags. Capturing the resting→state computed-style delta turns 'looks roughly right' into exact hover/focus/active fidelity in one shot.
- **Today:** `cfg.elementStates` is declared in `src/types.ts:347` and defaulted `true` in `src/config.ts:309`, but a full grep shows ZERO consumers — it is never read in pipeline.ts or any capture module. `src/capture/screenshot.ts` only does full-page shots; no pseudo-state driving exists. FIDELITY-AUDIT.md:127 lists 'element-state capture (hover/focus/active/disabled)' as still-open.
- **Target modules:** src/capture/screenshot.ts, src/capture/elementStates.ts (new), src/pipeline.ts, src/explore/clickables.ts, src/output/bundle.ts

#### Pierce Shadow DOM + web components in DOM, token, and clickable capture  _(impact: high · effort: M)_

- **Problem:** Sites built on web components (Lit, Stencil, FAST, Salesforce LWC, many design-system widgets, and increasingly Shopify/embedded widgets) render their real markup and styles INSIDE shadow roots. Every capture path here uses light-DOM-only APIs, so the agent receives an empty/placeholder shell for those components: no inner structure, no per-component encapsulated CSS, no design tokens from inside the shadow tree, and the explorer cannot click controls living in shadow roots.
- **Proposal:** Add shadow-piercing to the in-page evaluators. (1) In `captureDom`, after `page.content()`, run a `page.evaluate` that walks the tree, and for every element with `.shadowRoot` (open), serialize its `shadowRoot.innerHTML` plus its `adoptedStyleSheets` (via `cssRules`) into an inert `<template shadowrootmode="open">` declarative-shadow-DOM block — producing an HTML dump that actually round-trips the component tree and its scoped CSS. (2) In `extractTokens`/`extractCssVars`, recurse into `el.shadowRoot` children and into `adoptedStyleSheets` so encapsulated colors/fonts/`--vars` are tallied. (3) In `enumerateClickables`, traverse shadow roots when collecting candidates so the explorer can reach in-shadow buttons (Playwright locators already pierce open shadow DOM, so the selectors remain actuatable). Detect `customElements`-defined tags and list them in a `web-components.json` inventory so the prompt tells the agent which custom elements exist.
- **Why it matters:** For any web-component site this is the difference between capturing the real UI and capturing nothing. A one-shot rebuild from a hollow `<my-card></my-card>` is impossible; with the shadow tree + scoped CSS + token tally the agent can faithfully reconstruct the component.
- **Today:** `src/extract/dom.ts:58` is `page.content()` (a thin wrapper, light DOM only — its own comment notes it could be swapped). `src/extract/tokens.ts` and `src/extract/clickables.ts` walk only `document.querySelectorAll('*')` with no `.shadowRoot` recursion. A grep for `shadowRoot|attachShadow|adoptedStyleSheets|::part` across src returns ZERO matches.
- **Target modules:** src/extract/dom.ts, src/extract/tokens.ts, src/explore/clickables.ts, src/output/bundle.ts, src/prompt/generate.ts

#### Capture dynamic visual surfaces: <canvas>/<video>/WebGL/<iframe> as first-class artifacts  _(impact: high · effort: M)_

- **Problem:** Charts, maps, games, video players, animated WebGL hero sections, and embedded iframes (Stripe/Maps/YouTube/Calendly) are invisible to the current capture: a full-page PNG flattens them into one image with no signal about what they ARE, and the DOM dump contains an empty `<canvas>` or a cross-origin `<iframe>` whose contents `page.content()` never sees. The agent rebuilds these as blank boxes.
- **Proposal:** Add a `surfaces.json` artifact produced per page by an in-page scan that enumerates: every `<canvas>` (with its 2D vs WebGL context type, CSS size, and a `toDataURL()` thumbnail snapshot saved as a PNG so the chart's rendered output survives), every `<video>`/`<audio>` (src/poster/dimensions/controls — and save the poster image off the wire), every `<svg>` that contains `<animate>`/`<animateTransform>`/SMIL or CSS-animated children (flagged as animated), and every same-origin `<iframe>` (recurse via `page.frames()` to dump the child frame's DOM + tokens; cross-origin frames recorded as embed metadata: src host + size). For canvas-heavy chart libs, also sniff the global namespace (window.Chart / d3 / echarts / Highcharts / Plotly) and record the detected charting library. This gives the rebuild a typed inventory ('this box is an echarts line chart, 600x300, here's its rendered snapshot') instead of an empty element.
- **Why it matters:** Data-viz and embeds are frequently the centerpiece of dashboards and marketing pages — exactly the busy pages the tool targets. Without typing these surfaces and snapshotting canvas output, the most visually important regions become stubs, which is a hard fail for one-shot 1:1.
- **Today:** No canvas/video/iframe/WebGL handling exists anywhere: grep for `canvas|video|iframe|webgl|page.frames` across src/capture, src/extract returns nothing. `src/capture/screenshot.ts` does a single flattened `fullPage` PNG. `captureDom` (`src/extract/dom.ts:58`) cannot see cross-origin iframe content or canvas pixels.
- **Target modules:** src/extract/surfaces.ts (new), src/extract/assets.ts, src/pipeline.ts, src/output/bundle.ts, src/prompt/generate.ts

#### Capture loading / empty / error / skeleton states as first-class screenshot+DOM artifacts  _(impact: medium · effort: M)_

- **Problem:** The explorer already CLASSIFIES transient states (`sampleTransientState` returns loading/empty/error/success) but never captures them as standalone artifacts you can rebuild from. Resting full-page captures wait for networkidle and lazy-scroll, so they deliberately capture the settled, populated state — meaning skeleton loaders, empty-state illustrations ('No results yet'), and error banners are systematically excluded from the bundle even though they are real, designed UI states the twin must implement.
- **Proposal:** Two additions. (1) Synthetic state induction: for a sampled set of routes, capture under throttled/blocked network to freeze the LOADING/skeleton state — use Playwright route interception to delay or abort the page's XHR/fetch and screenshot the skeleton before data arrives; and induce the EMPTY state by driving a search/filter input with a no-match query (e.g. a random UUID) and capturing the resulting empty list. (2) Promote the explorer's already-detected transient states: when `sampleTransientState` returns loading/empty/error, save a dedicated screenshot + DOM + a11y golden tagged with the state in `behaviors.json`, instead of only stamping a string on the action record. Index these in `bundle.json` as `stateVariants` per route.
- **Why it matters:** Loading/empty/error are designed states with their own layout, copy, and illustrations; a rebuild that only handles the happy path is visibly incomplete and fails real QA. The detection logic already exists — this closes the gap between 'we noticed it' and 'we captured it so it can be rebuilt'.
- **Today:** `src/explore/engine.ts:327-367` `sampleTransientState()` detects loading/empty/error/success and stamps `transientState` on the ActionRecord (engine.ts:646), but no dedicated screenshot/DOM is saved for these states and they aren't induced on demand. `src/capture/prepare.ts` waits for networkidle + fonts, explicitly capturing only the settled state. FIDELITY-AUDIT.md:71 flags 'loading/empty/error not first-class'.
- **Target modules:** src/explore/engine.ts, src/capture/states.ts (new), src/pipeline.ts, src/explore/behaviors.ts, src/output/bundle.ts

#### Exact box-model / layout geometry capture for key components (computed spacing oracle)  _(impact: high · effort: M)_

- **Problem:** Design tokens today are FREQUENCY tallies of values (e.g. 'gap: 16px — 240 uses') with no association between a value and the element/region it applies to. The agent knows 16px is a common gap but not that the card grid uses 24px column-gap while the nav uses 8px. Spacing/sizing is the single most laborious thing to match by eye from a screenshot, and the current token list does not pin it to structure.
- **Proposal:** Add a `layout.json` per page: for a curated set of structural landmarks (header/nav/main/footer, the largest repeated card/list-item, primary buttons, form fields, modal/dialog) resolved via role + semantic selectors, record the computed box model — content/padding/border/margin (all four sides), width/height, display, flex/grid template (grid-template-columns, gap, flex-direction/justify/align), position + z-index, and font-size/line-height. Capture this for the primary breakpoint and, when `--breakpoints` is set, per breakpoint so responsive deltas are explicit. Emit a compact 'spacing contract' the REBUILD-PROMPT can cite ('card grid: display grid, 3 cols, 24px gap, 16px padding'). Keep it bounded to ~20-40 landmark elements per page to stay small.
- **Why it matters:** Pixel-exact spacing and grid structure are what separates a convincing twin from an approximation, and they're nearly impossible to recover accurately from a screenshot alone. Pinning real computed geometry to named landmarks gives the agent the numbers directly, dramatically cutting spacing guesswork in a one-shot build.
- **Today:** `src/extract/tokens.ts` tallies spacing/radii/shadows as global frequency counts only (`bump` into a flat `Record<string,number>`) with NO element/region association and NO per-breakpoint layout. There is no box-model, grid-template, or z-index/stacking capture anywhere in src/extract.
- **Target modules:** src/extract/layout.ts (new), src/extract/tokens.ts, src/pipeline.ts, src/prompt/generate.ts, src/output/bundle.ts

#### Deepen explorer enumeration so busy dashboards don't under-click (known 4-actions-on-a-busy-page gap)  _(impact: high · effort: M)_

- **Problem:** On dense dashboards the explorer records only a handful of actions. Three structural causes in the current code: (a) `enumerateClickables` caps the cursor:pointer scan at the FIRST 1500 DOM nodes (`scanLimit = Math.min(all.length, MAX_NODES)` over document order), so controls late in a long page are never even considered; (b) the structural-signature dedupe (`kind|normalizedLabel|structuralKey`) collapses genuinely distinct controls that share a tag/label shape (e.g. several icon-only buttons with empty labels in the same wrapper) down to one representative; (c) `maxActionsPerPage` default budgets cut off exploration before tabs/cards/filters deeper in the page are reached. Net: many in-page controls (tab strips, card actions, kebab menus, toolbar buttons) are silently dropped.
- **Proposal:** Make enumeration coverage-aware rather than first-N: (1) replace the fixed first-1500 cursor:pointer scan with a viewport-stratified scan (sample across the full scroll height so late-page controls are represented) and raise/segment the cap; (2) refine dedupe to keep a small quota PER distinct region (cluster by nearest landmark/container) instead of one global representative per signature, so a toolbar with 6 distinct icon buttons keeps all 6; (3) add an in-page 'interactive density' estimate to `run-manifest.json` (count of candidate controls vs. actions actually explored) so under-coverage is observable and the budget can be auto-raised for dense pages. Bound everything with the existing budgets but spend them on DIVERSITY (one per region/label-cluster) before depth.
- **Why it matters:** The whole value of `--full` is that the twin gets real features, not stubs. Systematically missing most controls on the densest, most feature-rich pages is precisely where one-shot fidelity collapses; this is the explicit 'only 4 actions on a busy page' gap.
- **Today:** `src/explore/clickables.ts:276-291` scans cursor:pointer over `document.querySelectorAll('*')` truncated to the first `MAX_NODES = 1500` in document order (line 13, 277). Dedupe at lines 374-376 uses one global `sigSeen` keyed `kind|normalizedLabel|structuralKey` keeping a single representative. The engine's `maxActionsPerPage` (engine.ts:169) caps before deep controls are reached. No interactive-density metric is recorded.
- **Target modules:** src/explore/clickables.ts, src/explore/engine.ts, src/output/manifest.ts

#### i18n/locale variant capture + print-stylesheet pass  _(impact: medium · effort: S)_

- **Problem:** Capture is pinned to a single locale (`locale=en-US`, deterministic default) and screen media only. Sites with locale switchers (RTL Arabic/Hebrew layouts, translated copy, locale-specific number/date formatting) and sites with meaningful print styles (invoices, receipts, articles with `@media print`) lose those entire visual axes. An RTL pass in particular changes the whole layout direction, which a screenshot in en-US can never reveal.
- **Proposal:** Two small, screenshot-only passes mirroring the existing dark/breakpoint variant machinery. (1) `--locales <list>`: re-capture key routes in an additional context with a different `locale`/`Accept-Language` and, when a locale switcher control is detected, actuate it; auto-flag RTL by reading `document.dir`/`getComputedStyle(html).direction` and capture the mirrored layout. (2) `--print`: emulate print media (`page.emulateMedia({ media: 'print' })`) and capture a print-styled screenshot/PDF for routes that ship a print stylesheet (detect `@media print` rules during the CSSOM walk so the pass only runs where it matters). Both reuse the isolated-context variant pattern so the primary flow is untouched and default runs do zero extra work.
- **Why it matters:** For internationalized and document-style sites these are real, designed presentations the twin must reproduce; RTL especially is a structural layout flip that's invisible to a single-locale capture. Reusing the proven variant pipeline makes it low-risk, high-coverage breadth.
- **Today:** `src/pipeline.ts:458-528` already implements isolated-context variant passes for breakpoints × color-scheme via `launchSessionForVariant`, but the only axes are viewport and `colorScheme`. Determinism pins `locale=en-US` (README:354) and nothing emulates print media (grep for `emulateMedia`/`media: 'print'` returns nothing). No RTL/direction detection exists.
- **Target modules:** src/pipeline.ts, src/capture/browser.ts, src/cli.ts, src/output/bundle.ts

### Angle: rebuild-quality

#### Visual-regression gate: pixel/perceptual diff of original capture vs rebuild (new `verify`/`vdiff` command)  _(impact: high · effort: M)_

- **Problem:** The tool has gates for a11y trees (Dice score) and data-fidelity (captured values render), but NOTHING that compares the rebuilt UI's PIXELS to the captured screenshots. A rebuild can pass a11y-diff and qc-tasks while looking visibly wrong (wrong spacing, broken layout, missing image, off colors) — the one signal a human grades on is the one with no automated gate. The README/FIDELITY-AUDIT explicitly call this out as missing ('no visual-regression (pixel/perceptual-diff) gate').
- **Proposal:** Add a `screenshotter vdiff <bundle> --target <url>` (and fold it into the unified verify command) that, for each route in bundle.json, launches the rebuild at the same path + the SAME deterministic context already used for capture (frozen clock, reduced-motion, anim-disabled — `src/determinism.ts`/`src/capture/browser.ts`), takes a full-page screenshot at the captured viewport, and computes a perceptual diff against the bundle's PNG using pixelmatch+pngjs (small, zero-native deps). Emit a per-route mismatch ratio + a diff PNG, and a threshold-based pass/fail with non-zero exit (CI gate). Reuse `routeUrlFor` from `src/qc/run.ts` for path re-basing and `planCaptures`/`captureScreenshot` for parity. Mask the same `--mask` regions and dynamic areas so the diff compares signal not noise (the determinism layer already advertises this readiness).
- **Why it matters:** Closes the single largest verification hole for a 1:1 VISUAL twin. Determinism is already engineered specifically to make pixel diffs meaningful, so the substrate exists — only the comparator and gate are missing. Gives the agent (and CI) an objective 'how close am I visually' number to iterate against in the one-shot loop.
- **Today:** No pixel/perceptual diff exists anywhere — `grep` for pixelmatch/pngjs/odiff/perceptual returns nothing; no such devDependency in package.json. `src/determinism.ts` comments say captures are made reproducible 'so visual-diff compares signal not noise', but no consumer of that exists. #### Multi-step USER FLOW capture → replayable flow scripts (checkout/onboarding/search-to-detail)  _(impact: high · effort: L)_

- **Problem:** The bundle teaches the twin what each control does in isolation, but not what a coherent end-to-end task looks like. A rebuild can pass every QC task (each modal opens, each search fires) yet still have a broken checkout or onboarding because the multi-step SEQUENCE — fill A, submit, land on B, fill C, confirm, see success — was never recorded as a unit. This is the difference between 'all the parts work' and 'the app works'.
- **Proposal:** Add a `flows` artifact: a `--flows` pass (or LLM/heuristic chaining over existing exploration data) that stitches atomic actions into ordered, named, replayable flow scripts. The substrate already exists — `explore/engine.ts` records every action with a `path: Clickable[]` precondition and a `reach()` replayer that re-navigates + replays fill/select/click steps. Extend it to (a) follow a navigation outcome into the next page and KEEP recording (instead of resetting to baseline each action), producing chains like `search → results → open detail → open 'Use this model' → copy snippet`; (b) in `--aggressive`, record genuine submit chains (signup form → submit → confirmation) that `interact.ts`/`safety.ts` currently hard-skip; (c) emit `explore/flows.json` as ordered semantic steps (role+name+value+expected-state per step) plus a per-flow result-state screenshot reel. Then teach `qc/run.ts` a `flow` task kind that replays the whole chain against the rebuild and scores it as one pass/fail, and surface flows in REBUILD-PROMPT.md as first-class 'user journeys to implement'.
- **Why it matters:** Journeys are the actual product. A one-shot rebuild that reproduces per-page states but not the connective tissue between them is functionally incomplete — and today nothing in the bundle even names the journeys, so the agent has to infer them from a flat behavior list. Flow scripts make the highest-value features (checkout, onboarding, search-to-purchase) buildable AND verifiable in one shot.
- **Today:** explore/engine.ts records ONLY atomic actions: each click is preceded by `reach(pathArr)` which re-navigates to baseline and replays the path, then it explores one delta and the DFS recurses, but navigation outcomes start a fresh selector namespace rather than continuing one flow (engine.ts lines 654-671). behaviors.ts collapses everything to single-trigger `features` + single-edge `routes` (no ordered chain). qc/run.ts runOne replays exactly one control with optional `pre` steps — there is no multi-page flow task kind. interact.ts + explore/safety.ts hard-skip submit/checkout/confirm so real flows are never traversed. No flows.json, no journey concept in prompt/generate.ts.
- **Target modules:** src/explore/engine.ts, src/explore/behaviors.ts, src/types.ts, src/qc/generate.ts, src/qc/run.ts, src/prompt/generate.ts
